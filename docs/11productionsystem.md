# Branz.Idle Production Calculation System

---

# 1. Production Engine Philosophy

In adherence to our **Progress Never Stops** philosophy, resource generation is purely **Time-Based (Timestamp Arithmetic)** rather than tick-dependent simulation.

Whether a player is actively running around their territory inside the dedicated `idle_world` or offline for 18 hours while sleeping, the core mathematical yield formula is identical. The server never spawns heavy per-tick tasks for every node; instead, it evaluates elapsed time intervals against `last_calculated_time`.

---

# 2. Online Production Ticker (`ProductionService`)

To simulate live production while a player is online, a lightweight Bukkit scheduler (`ProductionTickerTask`) runs at a configurable interval (`e.g., every 5 seconds`):

```text
Bukkit Scheduler (Runs every 5s)
 └── For each active Node in NodeCache:
      ├── Calculate elapsed = System.currentTimeMillis() - node.lastCalculatedTime
      ├── Check if elapsed >= effectiveTickIntervalMillis
      ├── If true -> Calculate produced cycles & add to NodeStorage buffer
      └── Update node.lastCalculatedTime & fire ResourceProducedEvent
```

## 2.1 Effective Interval Calculation
A node's base tick interval (`baseTickSeconds` from `nodes.yml`) is accelerated by the summed `productionSpeedBonus` of all assigned workers:
$$\text{EffectiveIntervalSeconds} = \frac{\text{BaseTickSeconds}}{1 + \sum \text{WorkerSpeedBonus}}$$

If `BaseTickSeconds = 60s` and two assigned workers provide a combined `+50% speed bonus` (`1.50`), the new interval becomes exactly `40 seconds`.

---

# 3. Offline Catch-Up Delta Calculation

When a player reconnects (`PlayerJoinEvent`) or enters an unloaded territory chunk, `ProductionService.calculateOfflineDelta(node)` executes asynchronously on the background thread pool:

## 3.1 Offline Duration Cap
To prevent infinite resource hoarding after months of player inactivity from destabilizing the economy, the total elapsed offline time (`deltaMillis`) is bounded by a configurable cap (`config.yml` -> `production.offline_max_hours`):

```java
long rawDeltaMillis = System.currentTimeMillis() - node.getLastCalculatedTime();
long maxAllowedMillis = TimeUnit.HOURS.toMillis(config.getLong("production.offline_max_hours", 24));
long effectiveDeltaMillis = Math.min(rawDeltaMillis, maxAllowedMillis);
```

## 3.2 Catch-Up Execution Flow
1. **Calculate Ticks**:
   $$\text{CompletedCycles} = \lfloor \frac{\text{effectiveDeltaMillis}}{\text{EffectiveIntervalMillis}} \rfloor$$
2. **Calculate Yield**:
   $$\text{TotalYield} = \text{CompletedCycles} \times \text{BaseOutputQuantity} \times (1 + \sum \text{WorkerYieldBonus})$$
3. **Roll Exploration Drop Tables**:
   * For each completed cycle (`or scaled batch`), `ProductionService` queries `DropTableRegistry` using the node's current `explorationLevel`.
   * Assigned worker `rareDropBonus` multipliers are applied to increase the weights of `RARE`, `EPIC`, and `LEGENDARY` drop table entries.
4. **Enforce Storage Capacity**:
   * All calculated standard resources and rare exploration drops are added to the node's local buffer (`NodeStorage`).
   * If total quantity reaches `maxCapacity` (`defined by node level`), generation halts cleanly (`buffer full`).
5. **Worker XP Distribution**:
   * All assigned workers (`WorkerInstance`) receive proportional accumulated XP for the `CompletedCycles`, allowing them to level up automatically while the player was offline.
6. **Publish Event**:
   * `ResourceProducedEvent` (`isOfflineCatchUp = true`) is fired, and `SaveQueueService` persists the updated storage buffer and worker XP levels to the database asynchronously.

---

# 4. Collection Flow (`NodeStorage` -> `PlayerResourceWallet`)

Resources generated by online ticks or offline catch-up are **never** deposited directly into the player's physical Minecraft inventory (`which would overflow immediately`).

```text
+-------------------------------------------------------------+
|                      NODE STORAGE BUFFER                    |
| * Stores produced ore, wood, or fish up to max capacity     |
| * Example: { iron_ore: 4500 / 5000, coal: 1200 / 5000 }     |
+-------------------------------------------------------------+
                               |
           Player clicks "Collect All" inside Node GUI
                               v
+-------------------------------------------------------------+
|                 PLAYER VIRTUAL RESOURCE WALLET              |
| * Persistent player_resource database table                 |
| * Infinite stacking, safely holds hundreds of thousands     |
| * Used directly as currency/materials for Node Upgrades     |
+-------------------------------------------------------------+
```

## Lifecycle of Resource Collection (`StorageService.collectNodeStorage`)
1. Player opens Node GUI (`or Base Dashboard GUI`) and clicks **"Collect Resources"**.
2. `StorageService` verifies that `node_storage` contains items (`quantity > 0`).
3. All buffered resource entries (`resource_key, quantity`) are atomically transferred into the player's centralized `PlayerResourceWallet` (`player_resource` table).
4. `node_storage` quantities are reset to `0`, freeing up `100% buffer space` for the node to continue generating.
5. `ResourceCollectedEvent` is fired, and `SaveQueueService` updates both `node_storage` and `player_resource` records in batch.
